Windows

Unlike Linux, Windows bundles all those components together(by Windows and component version), which makes test planning simpler. But, that means that you need to test across many versions of Windows.

Unfortunately, testing many Windows versions can get expensive, so I've broken down the Windows versions to focus on depending on the type of application and how thorough you want to be.

Disclaimer: None of this information official Microsoft information. Rather this is based on my experiences in Posix]+Windows interoperability (also Older-Windows + Newer-Windows).

Key
Minimum Before you can say your software works on Windows you should test on each of these. If you don't test on a Windows or Windows Server version, you should NOT say it works on that version.
Client-Minimum For desktop applications, these are the Windows versions you should focus on. If possible, you should also test it on versions of Windows Server just in case some of your users use it on those.
Multimedia Multimedia applications may behave differently on x86 and x64. Also, you should also test your application on each Windows Server edition (2003|2003r2, 2008, 2008r2) because audio and video software(ex: VoIP Phone client) may behave differently on them (due to driver differences and security features).
Server-Minimum For Server software (ex: a web server), these are the Windows versions you should focus on.
Recommended For any software you want to run on Windows, you should really test on these versions.
Thorough To be through, to be really sure it'll work, test on these.

Note: run 'Winver' (type into Start/Run or Start/Search) to tell which Windows version is running.

Note: I use SP0 to refer to the RTM release, the very first release/version (simpler to use one notation for the service pack, the standard SPn notation).

Total Memory 6.8 GB 9.5 GB 14.2 GB 12.3 GB 23 GB 43.4 GB
  • This amount of memory will give your application about 3/4 of the system's memory within with to run (Windows will take the other 1/4).
  • For higher memory applications, you may want to increase the memory for the 'Pre-Longhorn' systems (2003, XP).
  • 43.4 + .5 GB for Debian, Ubuntu, CentOS, Fedora, Gentoo and FreeBSD(32 guests) => 46.4 GB Total to test your application across multiple platforms.
  • If you're only testing for functionality and don't care about performance, you can host those on just a few or even one, virtualization server.

You should test x86 code on x64 also, because Windows x64 uses an extra subsystem(Wow64) to support x86 which will at least, add a performance penalty to your application (though x64 CPUs tend to be more powerful than x86 which can compensate).

Before you can really say your software works on an OS, you need to have tested it on at least the minimum service pack(s) of that OS. It sounds obvious, but many companies don't do it.

If you're going to say 'It works on Windows' that includes Windows XP. You'll need to test on Windows XP(above). To avoid that, you should instead say 'It works on Windows Vista|2008 and newer'.

Past and present Windows OSes. Note: except for Azure, none of these are cross-platform and Azure is the only one of these variants supported by PFTT.

  • Generally, if it runs on Windows 2008 or 2008r2, it should work on Windows Azure.
